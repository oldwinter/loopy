Agent loop 如何工作

Agent loop 是带检查的任务。agent 做一些工作，检查结果，然后继续或停止。

实际上，agent 会观察当前状态，采取一个动作，检查发生了什么，并决定下一步怎么做。 当一步的结果应该改变下一步时，使用 loop。如果不会，就使用一次性任务。

什么让 loop 有用

从清晰目标开始。 用可以测量或审阅的方式描述结果。 “改进代码”很模糊。“在相同测试条件下，让每个页面在 50 毫秒内加载” 会给 agent 一个真实终点。

让每个动作保持小。 要求 agent 每次只做一个有边界、可逆的改动。 小改动更容易验证，也更容易在结果变差时撤销。

使用固定检查。 每次改动后运行同一个测试、benchmark、rubric 或批准步骤。 决定工作是否变好的是检查，而不是 agent 的看法。

定义如何停止。 说明成功是什么样子、什么时候不需要改动、什么时候应该请求批准， 以及什么时候因为阻塞或预算耗尽而停止。

目标 loop 手动开始，并持续运行到检查通过或预算耗尽。计划 loop 在固定时间或事件后启动。 每次计划运行仍然应该完成、报告结果，并等待下一次触发。

编写提示词

用普通语言编写 loop。写明触发条件、agent 应检查的当前信息、它可以做出的改动、 它必须运行的检查，以及让它停止的条件。

复制并改写这个模板：

当 [trigger] 时，检查 [fresh inputs]。使用 [criteria] 选择一个范围内动作，然后执行改动。

在相同条件下运行 [acceptance check]。在 [state file] 中记录改动内容、证据和下一步。

只有在进展可测量且 [budget] 仍然有效时重复。通过 [success gate] 时停止。 当 [no-op condition] 为真时，不做改动并停止。

当 [escalation condition] 出现时请求批准或报告阻塞。绝不执行 [forbidden action]。 最后产出 [pull request, report, artifact, or handoff]。

在设为计划任务前，先手动运行一次提示词。第一次运行通常会暴露缺失检查、 不清晰边界，或需要更具体的停止条件。

在你的编码 agent 中运行

同一个 loop 提示词可以跨工具使用。主要区别在于你把重复指令保存在哪里， 以及如何安排未来运行。

Cursor

打开仓库，选择 Agent，然后粘贴 loop 提示词。包含范围内文件、要运行的精确检查， 以及停止规则。审阅终端输出和 diff，然后用检查结果，而不是 agent 总结，判断工作是否完成。

对于重复工作，把稳定指导保存在 .cursor/rules 或 AGENTS.md。 使用 Cursor Automations 处理计划或事件触发运行。参见官方文档： Agent、 Rules 和 Automations.

Codex

打开仓库；如果 loop 应在隔离环境中改代码，启动 Worktree thread。粘贴包含目标、约束、 检查和“done when”条件的提示词。接受工作前，审阅 diff 以及最终测试或 benchmark 结果。

对于重复工作，把项目规则保存在 AGENTS.md。先手动测试提示词， 再创建 Codex Automation，并审阅前几次运行。参见官方文档： 最佳实践、 AGENTS.md 和 Automations.

Claude Code

运行 cd my-project，再运行 claude。粘贴包含检查、预算和停止规则的 loop 提示词。如果想生成初始 CLAUDE.md 作为稳定项目指令，可运行 /init。

在打开的 session 内轮询时，可以使用 /loop 5m check whether the deploy finished 这样的提示词。 对于必须在终端关闭后继续存在的工作，使用 Routine、Desktop task 或 GitHub Action。 参见官方文档： CLAUDE.md, scheduled tasks 和 Routines.

Factory

打开仓库，运行 droid，并粘贴 loop 提示词。包含工作目录、检查、最终输出和停止规则。 审阅 spec、权限、编辑和验证输出。

对于重复工作，从 CI 或 cron 调用 droid exec --auto medium -f .factory/prompts/loop.md。 使用能够完成任务的最低自治级别。参见官方文档： Droid CLI 和 Droid Exec.

Devin

使用正确仓库和分支启动 session。粘贴 loop 提示词，并说明最终输出应是 pull request、报告还是 clean no-op。 合并前审阅计划、改动和证据。

对于重复工作，把提示词保存为 Playbook。使用 Schedule Devin 或 Settings > Schedules 进行重复运行，然后审阅 Past Sessions。参见官方文档： Playbooks 和 Scheduled Sessions.

产品控件会变化。上面的链接都指向各产品官方文档。

保持 loop 安全

设置限制。 给每个 loop 一个最大时间、成本、重试次数、迭代次数或影响范围。 loop 绝不应把“继续”理解为无限权限。

保持检查稳定。 每次结果后都改 benchmark 会让进展无法比较。 当 loop 在优化提示词或模型时，使用新鲜的验收集。

留下有用交接。 在 tmp/<file>.md 中记录目标、已完成步骤、 证据、阻塞和下一步。不要把秘密存在那里。

重大动作必须经过批准。 生产、破坏性、财务、隐私敏感或对外消息动作需要人来批准。 blocked、exhausted 和 stagnant 的运行不是成功运行。
